빠른 시작

Snowflake Cortex 번역 서비스

이 저장소는 로컬 CLI와 FastAPI REST API를 통해 Snowflake Cortex 번역 기능을 제공합니다. 설정, 실행, 배포 흐름을 한 번에 정리해 팀이 바로 적용할 수 있는 운영 기준과 예시를 제공합니다.

기본 연결

dev Snowflake 프로필

주요 진입점

CLI + REST API + Docker

운영 기준

로컬 실행 + 컨테이너 배포

01

프로젝트 개요

무엇을 제공하나요?

  • 명명된 Snowflake 연결 프로필 활성화
  • AI_TRANSLATE 호출을 통한 번역 처리
  • CLI와 REST API 양쪽 인터페이스 제공
  • 운영·배포·보안 가이드를 저장소 문서로 관리

구성 요소

도메인
번역 요청/응답 모델과 입력 검증
서비스 계층
게이트웨이 호출과 오류 매핑
전달 계층
CLI, FastAPI API, Docker 런타임

런타임 흐름

  1. 운영자가 SNOWFLAKE_HOME 아래 로컬 프로필을 준비합니다.
  2. CLI 또는 REST API가 번역 입력을 수집하고 검증합니다.
  3. 서비스 계층이 Snowflake 게이트웨이를 통해 AI_TRANSLATE를 실행합니다.
  4. 결과는 표준 출력 또는 JSON 응답으로 반환됩니다.

02

Snowflake 프로필 설정

사전 조건

  • Python 3.12+
  • uv
  • SNOWFLAKE.CORTEX_USER 권한이 있는 역할
  • 접근 가능한 warehouse / database / schema

인증 기본값

로컬 개발에서는 저장소 안에 비밀번호를 남기지 않기 위해 externalbrowser 사용이 가장 안전한 기본값입니다.

설정 절차

mkdir -p .snowflake
cp config/connections.toml.example .snowflake/connections.toml

export SNOWFLAKE_HOME="$PWD/.snowflake"
export SNOWFLAKE_DEFAULT_CONNECTION_NAME=dev
export SNOWFLAKE_CONNECTION_NAME=dev

.snowflake/connections.toml의 placeholder 값을 실제 계정, warehouse, database, schema 정보로 교체한 뒤 사용합니다.

실환경 검증

권한이 준비되면 sql/setup.sql과 sql/verify.sql을 사용해 Snowflake 쪽 권한 및 함수 호출 경로를 확인하세요.

03

CLI 사용법

의존성 설치

uv sync --dev

기본 번역

uv run cortex-translate \
  --text "Hello world" \
  --source en \
  --target ko

원문 언어 자동 감지

uv run cortex-translate \
  --text "Voy a likear tus fotos en Insta." \
  --target en

추적용 Query Tag 지정

SNOWFLAKE_QUERY_TAG=cortex-translate-demo \
uv run cortex-translate \
  --text "Good morning" \
  --source en \
  --target de

CLI 사용 팁

  • 번역 결과는 표준 출력으로 반환됩니다.
  • 실패 시에는 고수준 오류만 노출되며 내부 비밀값은 반환하지 않습니다.
  • 운영 추적이 필요하면 Query Tag를 함께 남기세요.

04

REST API 사용법

로컬 서버 실행

export TRANSLATION_API_KEY=change-me-before-exposing-the-api
uv run uvicorn cortex_translate_service.api:app \
  --host 127.0.0.1 \
  --port 8000
주요 엔드포인트
메서드 경로 설명
GET /healthz 애플리케이션 헬스 체크
POST /api/v1/translations 번역 요청 처리
GET /openapi.json OpenAPI 스키마 확인

헬스 체크

curl http://127.0.0.1:8000/healthz

번역 요청

curl -X POST http://127.0.0.1:8000/api/v1/translations \
  -H "Content-Type: application/json" \
  -H "x-api-key: $TRANSLATION_API_KEY" \
  -d '{
    "text": "Hello world",
    "source_language": "en",
    "target_language": "ko"
  }'

운영 메모

API는 요청/도메인/게이트웨이 오류를 제어된 JSON 응답으로 매핑합니다. 상세 내부 정보나 비밀값을 응답에 그대로 노출하지 않도록 설계되어 있습니다.

TRANSLATION_API_KEY가 설정되지 않으면 번역 엔드포인트는 503 응답으로 비활성화됩니다. 공개 배포 시에는 추가적인 네트워크 제어와 rate limiting을 함께 두세요.

05

Docker 사용법

이미지 빌드

docker build -t cortex-translate-service:local .

컨테이너 실행

docker run --rm -p 8000:8000 \
  -v "$PWD/.snowflake:/run/secrets/snowflake:ro" \
  -e SNOWFLAKE_HOME=/run/secrets/snowflake \
  -e SNOWFLAKE_DEFAULT_CONNECTION_NAME=dev \
  -e SNOWFLAKE_CONNECTION_NAME=dev \
  -e TRANSLATION_API_KEY="$TRANSLATION_API_KEY" \
  cortex-translate-service:local

주의 사항

  • Snowflake 연결 정보는 읽기 전용 마운트 또는 workload identity로 주입합니다.
  • 자격 증명을 이미지 안에 bake-in 하지 않습니다.
  • 릴리스 워크플로는 linux/amd64, linux/arm64 이미지를 대상으로 합니다.

06

기여 · Git Flow · 릴리스 흐름

브랜치 운영 원칙

  • develop에서 통합하고 main에서 릴리스를 관리합니다.
  • 짧게 유지되는 토픽 브랜치 사용: feature/*, release/*, hotfix/*, docs/*
  • 보호 브랜치에는 직접 push하지 않고 Pull Request를 통해 반영
  • 운영 영향이 있으면 PR 설명과 문서에도 함께 남김

권장 로컬 확인

uv run pytest --cov=src/cortex_translate_service --cov-report=term-missing
uv run mypy src tests
uv run interrogate src tests
docker build -t cortex-translate-service:local .

커밋/릴리스 규칙

  • Conventional Commit 접두사 권장: feat:, fix:, docs:, ci:, security:
  • CHANGELOG.md는 Keep a Changelog 형식 유지
  • main에서 검증된 커밋에 시맨틱 태그(예: vX.Y.Z)를 push하면 GHCR 릴리스 워크플로가 동작
  • 웹 요약 페이지와 저장소 문서를 함께 최신 상태로 유지합니다.

저장소 거버넌스

보호 브랜치, CodeQL, dependency review, OSSF Scorecard, CI coverage, GHCR 릴리스가 저장소 운영 기본선입니다.

07

보안 유의사항

비밀정보 관리

  • .snowflake/, 토큰, 로컬 환경 파일은 커밋하지 않습니다.
  • 평문 비밀번호보다 externalbrowser, OAuth, PAT, 키 페어, workload identity를 우선합니다.
  • API 오류 응답에 환경별 내부 정보를 담지 않습니다.
  • REST API는 x-api-key 인증과 요청당 5,000자 제한을 기본선으로 둡니다.

취약점 제보

  • 공개 GitHub Issue에 보안 취약점을 올리지 않습니다.
  • 가능하면 private GitHub security advisory를 사용합니다.
  • 영향 버전, 재현 단계, 완화 방향만 최소한으로 공유합니다.

응답 목표

보안 정책상 최초 확인 목표는 영업일 기준 3일, 1차 triage 목표는 7일입니다.
